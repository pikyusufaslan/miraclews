Why it exists

Most WS libraries trade your memory for their convenience.

Most WebSocket libraries are built around a task-per-connection model — one tokio::spawn per socket. At idle, that grows to roughly 3 KB of heap per connection before a single byte of application data is stored. A server holding 100k connections spends 300 MB just on runtime overhead.

Others pull in a framing crate and schedule work through a runtime's event loop, adding dispatch latency on every message and making it hard to reason about backpressure.

MiracleWS is a thin transport core — not a framework. You implement one Handler trait; MiracleWS owns the TCP lifecycle, the RFC 6455 framing, masking validation, and backpressure. There is no async runtime in the hot path. No Tokio. No spawn overhead. No shared scheduler in the critical loop.

The result is a server that holds 300k idle connections on a single box in under 30 MB of connection state — and saturates local loopback at over 2.5 M messages per second, p99 under 10 ms.

Architecture

One Handler trait. N isolated reactors.

MiracleWS uses a sharded reactor model. Each CPU core runs its own event loop — no shared mutable state in the I/O path.

No cross-shard coordination
Each shard owns its connections exclusively. There are no mutexes, no channels, and no shared queues in the I/O path. Shards never wait on each other.
Two backends, one protocol engine
Switch between raw libc epoll and io_uring (with or without registered buffer rings) via a compile-time feature flag. The RFC 6455 engine is shared.
Your Handler, called inline
There is no task queue or scheduler between the reactor loop and your code. The Handler is invoked directly inside the event loop — no indirection, no wakeup latency.

Measurements taken on loopback with shared CPU — both client and server on the same host. Real-world numbers will vary. Latency figures are at saturation; unloaded p99 is sub-millisecond in all backends.
